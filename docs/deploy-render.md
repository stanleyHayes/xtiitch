# Deploying Xtiitch on Render

The repo ships a Render Blueprint at [`render.yaml`](../render.yaml). It defines
the whole stack: Go **API**, BullMQ **worker**, four SSR web apps
(**marketing**, **storefront**, **dashboard**, **admin**), **Postgres**, and
**Redis (Key Value)**.

## 1. Create the blueprint

1. Push to GitHub (done — `render.yaml` is on `main`).
2. Render dashboard → **New** → **Blueprint** → pick this repo → **Apply**.
3. Render creates every service and prompts for the `sync: false` secrets below.

## 2. Plans / cost

| Resource | Plan | Why |
|---|---|---|
| Postgres | **Basic** (paid) | Free Postgres is deleted after 30 days |
| Worker | **Starter** (paid) | Render has no free worker tier |
| API + 4 web apps | **Free** | Sleep after ~15 min idle; cold-start on next request |
| Redis | **Free** | 25 MB, backs the notification queue |

> For real traffic, bump at least **xtiitch-api** to Starter so the web apps
> aren't blocked on a sleeping backend.

## 3. Secrets to set (Render dashboard → each service → Environment)

**API (`xtiitch-api`)** — required:
- `ADMIN_BOOTSTRAP_PASSWORD` — first owner login (with `owner@xtiitch.com`)
- `PAYSTACK_SECRET_KEY`, `PAYSTACK_PUBLIC_KEY`, `PAYSTACK_WEBHOOK_SECRET`
- `CLOUDINARY_URL` — image uploads
- `RESEND_API_KEY` — transactional email (domain xtiitch.com is verified)

**API** — optional (each has a safe key-free fallback):
- `OPENAI_API_KEY` — AI search uses the hosted embedder (else a dev embedder)
- `ANTHROPIC_API_KEY` — AI search query understanding via Claude (else heuristic)
- `WHATSAPP_VERIFY_TOKEN` + `WHATSAPP_APP_SECRET` — enable the inbound bot webhook
- `WHATSAPP_PHONE_NUMBER_ID` + `WHATSAPP_ACCESS_TOKEN` — bot reply sending

**Worker (`xtiitch-worker`)** — required to boot:
- `WHATSAPP_PHONE_NUMBER_ID` + `WHATSAPP_ACCESS_TOKEN`. The worker fails fast in
  production unless `NOTIFICATION_TRANSPORT` can actually deliver (it's set to
  `whatsapp_cloud`). Until you have WhatsApp Cloud credentials, either set them or
  pause/scale the worker to zero — otherwise it restart-loops. The rest of the app
  works without it; only async notifications (order confirmations, etc.) wait.

`JWT_SIGNING_KEY`, `MFA_ENCRYPTION_KEY`, and each app's `SESSION_SECRET` are
auto-generated by Render (`generateValue`) — no action needed.

## 4. External webhooks (after the API is live)

Point these at the API's public URL (default `https://xtiitch-api.onrender.com`):
- **Paystack** dashboard → Webhook URL → `…/v1/webhooks/paystack`
- **Meta WhatsApp** → Callback URL → `…/v1/webhooks/whatsapp`, Verify Token =
  your `WHATSAPP_VERIFY_TOKEN`.

## 5. Database migrations

The API runs `goose up` against `infra/migrations` on every boot (built into its
start command), so the schema is applied automatically.

**One caveat:** migration `000004_rls_app_role` runs `CREATE ROLE xtiitch_app`,
which requires the database user to have the `CREATEROLE` attribute. Render's
primary database user normally has it, so this just works. If the first deploy
fails with `permission denied to create role` / `must have CREATEROLE`, create
the role once via Render's PSQL (database → **Connect** → PSQL) before the next
deploy:

```sql
CREATE ROLE xtiitch_app LOGIN PASSWORD '<choose-a-strong-password>'
  NOSUPERUSER NOBYPASSRLS NOCREATEDB NOCREATEROLE;
```

(The migration is idempotent — it skips creation if the role already exists.)

> **Tenant isolation note:** the API connects with Render's managed (owner)
> credentials. That is safe here because **every** RLS-enabled tenant table also
> has `FORCE ROW LEVEL SECURITY`, so the database enforces row-level security even
> for the table owner. The dedicated low-privilege `xtiitch_app` role is a
> defense-in-depth hardening you can switch to later by pointing the API's
> `DATABASE_URL` at that role with a strong password.

## 6. Cross-service URLs

`render.yaml` hard-codes the default `https://<service>.onrender.com` origins
(API URL for the web apps, CORS allow-list + dashboard/storefront URLs for the
API and marketing). If you attach **custom domains**, update those env values to
match.

## 7. First login

Once the API is healthy, sign in to the admin console
(`https://xtiitch-admin.onrender.com`) with `owner@xtiitch.com` and the
`ADMIN_BOOTSTRAP_PASSWORD` you set. The owner is bootstrapped on API start.
